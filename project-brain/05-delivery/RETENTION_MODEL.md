# Retention Model

## Purpose

Retention is about keeping operational systems useful, safe, and aligned with the client business after delivery. This document avoids pricing assumptions and focuses on the operating model.

## Retention fit

A client is a good fit for ongoing optimisation when:

- The delivered workflow is used regularly.
- The business expects process changes over time.
- Users generate feedback that should be reviewed and prioritised.
- There are multiple future automation opportunities.
- Access, security, and documentation need periodic review.

## Retention stages

## 1. Post-handoff review

### Humans do

- Check whether users understand the workflow.
- Review early issues and support questions.
- Decide which fixes are urgent and which belong in a backlog.

### Agents do

- Summarise support notes and user feedback.
- Group issues into fixes, improvements, training gaps, and new ideas.

### Automation happens

- Feedback can be categorised.
- Follow-up tasks can be drafted for human approval.

### Deliverables

- Post-handoff review note.
- Early issue list.
- Training or documentation updates.

## 2. Operational review cycle

### Humans do

- Review whether the system still matches the business process.
- Confirm new risks, access changes, or workflow changes.
- Prioritise improvements with the client.

### Agents do

- Prepare review summaries from approved notes and metrics.
- Identify repeated support patterns or bottlenecks.
- Draft improvement options.

### Automation happens

- Review reminders and checklists can be standardised.
- Approved operational metrics can be summarised.

### Deliverables

- Review summary.
- Updated improvement backlog.
- Risk and access review notes.

## 3. Continuous improvement backlog

### Humans do

- Decide which backlog items are worth doing.
- Confirm scope for each new increment.
- Approve changes before work begins.

### Agents do

- Keep backlog items structured.
- Draft small task briefs.
- Flag dependencies, risks, and missing source material.

### Automation happens

- Backlog formatting and task creation can be standardised.
- Repeated requests can be grouped into themes.

### Deliverables

- Prioritised backlog.
- Approved next task brief.
- Deferred ideas list.

## 4. Maintenance and governance

### Humans do

- Review access permissions.
- Confirm documentation remains accurate.
- Approve dependency, integration, or workflow changes.
- Decide when a system should be simplified, replaced, or retired.

### Agents do

- Draft maintenance checklists.
- Compare documentation against approved changes.
- Highlight stale assumptions or missing handoff notes.

### Automation happens

- Maintenance reminders can be prepared.
- Documentation gaps can be tracked.
- Standard checks can be repeated where approved.

### Deliverables

- Maintenance checklist.
- Updated documentation.
- Access and security notes.
- Change recommendations.

## Retention boundaries

- No pricing model is defined here.
- No service-level commitment is assumed.
- No specific tooling or monitoring stack is selected.
- New implementation work requires an approved task.
- Sensitive client data remains subject to explicit access approval.
